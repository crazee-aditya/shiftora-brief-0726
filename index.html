The backtest protocol, published in full
  • Holdout, frozen at kickoff. The most recent 12 months of history are sealed before any model work begins; nothing ever trains on them.
  • Rolling-origin evaluation. Each holdout month is forecast from an origin that sees only earlier data; the origin rolls forward and repeats. Twelve origins, twelve out-of-sample tests.
  • Accuracy and bias, gated together. Accuracy is reported as 100 minus WAPE; bias is gated at plus or minus 3 percent alongside it, so accuracy cannot be achieved through systematic over- or under-forecasting.
  • Baseline first. The incumbent method is scored under the identical protocol before ours; the claim is the difference, same data, same window, same math.
  • Re-proven every cycle. The backtest re-runs on every retrain and rule change. Any figure that fails re-validation is withdrawn.

The analyses that can invalidate the inventory

The initial analyses are selected to falsify the thesis rather than to support it.

  • Exception recurrence from the logs, not the interviews. If the top 11 types are not actually recurring, auto-resolution value collapses and the inventory is revised.
  • Hours rebuilt from tickets and system timestamps, cross-checked against what interviews claimed. Interview estimates typically overstate; system timestamps do not.
  • DSO slippage by customer cohort against contractual terms. If slippage concentrates in three accounts, the remedy is collections process rather than automation.
  • Booked rules versus actual postings. The gap between the SOP and reality is measured before anything is automated against either.

How agents earn a release
  • Evaluation environments: simulated replays of historical operational states; the agent must beat the human baseline inside the environment before it touches production.
  • Golden datasets: frozen, human-labelled sets per task; every prompt, policy, or model change runs the full regression suite.
  • Canary → shadow → full: new versions run in shadow against live traffic, diffed against the incumbent, promoted only on non-regression.
  • Drift monitors: input distributions and confidence calibration tracked continuously; drift beyond threshold pages an engineer, not a dashboard.
  • Version pinning: model, prompt, policy, and rule-engine versions pinned per decision, so any past output is reproducible on demand.

SOPs, formats, and vendors move. Drift is watched, reported monthly, absorbed as rule updates, human sign-off wherever mapping is ambiguous.
